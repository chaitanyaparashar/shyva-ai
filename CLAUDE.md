# CLAUDE.md

## Project: Supplier Intelligence Engine (Take-Home Evaluation)

This is a **time-boxed architecture + coding evaluation**, not a production build. The graders are explicitly assessing three things, weighted:

- **System Architecture (45%)** — data pipelines, LLM orchestration, RAG, pragmatic stack choices for an early-stage startup.
- **Algorithmic Thinking (30%)** — entity normalization that elegantly handles aliases, parent/subsidiary relationships, and conflicting field values.
- **Product Judgment (25%)** — sharp prioritization under a 2-hour constraint and honest recognition of where the design breaks.

**Optimize for signal, not completeness.** A working end-to-end app is explicitly *not* expected. A crisp architecture, one genuinely well-reasoned normalization module, and a ruthless decision log beat a sprawling half-built system.

**Prototype first, vision on top.** The deliverable is a thin runnable prototype that proves the thinking, wrapped in docs that project the full production/scalability vision. Exactly **one artifact runs** — `normalize.py`, the entity-resolution prototype on mocked data. Everything else (crawlers, vector store, orchestrator, serving API) is *described*, not built — describing infrastructure is correct and expected; do not try to stand any of it up. Every artifact must carry a **dual signal**: the prototype-scale thing actually built, *and* exactly how it scales to production (the "ship at launch vs. scale to" seam). Knowing precisely where the prototype ends and production begins is the senior signal.

**Build order (protects the score under time pressure):** (1) `normalize.py` first — the only thing that must run, 30% of the grade; (2) `ARCHITECTURE.md` second — 45%, breadth over depth, where the scaling vision lives; (3) `DECISION_LOG.md` last — 25%, fast to write, reflects on the first two. If time runs short, protect this order: a runnable+verified `normalize.py` plus a complete decision log outrank an exhaustive architecture doc. Never ship an unrun `normalize.py`.

## The Test Query (anchor every design decision to this)

> "Lithium-ion battery cell manufacturers in Japan or South Korea that supply to automotive OEMs, with ISO-9001 or IATF-16949 certification and an established export presence."

Notice this query contains: a **product/category** filter, a **geography** filter (disjunction), a **relationship** filter (supplies-to automotive OEMs), a **certification** filter (disjunction), and a **fuzzy/qualitative** filter ("established export presence"). The architecture must show how each of these decomposes into structured constraints and how confidence is scored *per field*, since some constraints are hard facts and others are inferred.

## Deliverables (exactly three)

1. **System Architecture Sketch** — markdown + a diagram (Mermaid preferred so it renders). End-to-end: query parsing → retrieval/RAG → per-field confidence scoring → serving. Must explicitly address the **latency vs. accuracy** tradeoff.
2. **Core Normalization Logic** — a single focused, runnable Python module. Mock the input as a messy JSON array of supplier records with conflicting revenue, legal-name aliases (e.g. `CATL` vs `Contemporary Amperex Technology Co. Limited`), and parent/subsidiary noise. Output one clean canonical record per real-world entity. **This is where the algorithmic 30% lives — it should be the most polished artifact.**
3. **Decision Log** — one tight paragraph: what was prioritized given 2 hours, the biggest architectural vulnerabilities, and the single first thing to build with a full week.

## LLM Governance (applies to the whole design)

Treat models as a **recall-and-language engine, not a source of truth**, and use them by the principle of **minimal sufficient usage**: the smallest model that clears a stage's accuracy bar, a model added to a stage only where cheaper deterministic logic can't do the job, and non-LLM models (an embedding model for blocking, a cross-encoder for re-ranking) wherever they're cheaper and better than a chat LLM. Models belong at the two ends of the pipeline — language-in (query decomposition, constraint typing, unstructured→structured extraction) and language-out (RAG synthesis, re-ranking, conversational refinement). The **HIGH-risk core is deterministic by mandate and the LLM is excluded from it**: relationship truth ("supplies to automotive OEMs") and field survivorship (which revenue/country/cert becomes canonical) are never picked or generated by an LLM — they come from cited structured sources or deterministic policy only. Merge/hierarchy is the one middle case: the LLM may *propose* into a staging area with a confidence score, and a **second, different-family model acts as an independent cross-checker on that proposal** (agreement raises confidence, disagreement routes to human review) — this is the *single* place a second model is justified, because it's low-volume and high-stakes. Deterministic rules always dispose; a model never autocommits a merge. Every model output carries provenance and a confidence value; anything without a cited source span or a deterministic gate does not become a "verified" fact. Multi-model routing and the cross-checker are **architected in but shipped single-family at launch** — they're switched on only once the evaluation set makes their accuracy gains measurable. The final `ARCHITECTURE.md` must state the HIGH-risk exclusion and the minimal-usage stance explicitly.

## Standards & Constraints

- **Language:** Python for the normalization module (stdlib + `rapidfuzz` only; do not require a network call or an LLM API key to run it). Keep it runnable with `python normalize.py`.
- **The normalization code must actually run** and print a before/after. Graders may execute it. Include a `if __name__ == "__main__"` demo with the mocked messy data baked in.
- **Show the reasoning, don't just assert it.** Every non-obvious tradeoff (why blocking before fuzzy match, why a canonical-name authority, why a confidence score instead of a boolean) gets one sentence of justification inline or in the log.
- **Be honest about failure modes.** The decision log is graded on recognizing where the architecture breaks, so name real vulnerabilities (hallucinated relationships from the LLM, stale cert data, alias collisions between distinct entities, no ground-truth to evaluate against).
- **Pragmatic > trendy.** Justify each stack choice as if defending a burn rate, and **name concrete tools with a "ship at launch vs. scale to" split**. Default early stack: Postgres + pgvector (one datastore, not a separate vector DB), Redis (cache + queue + rate-limiting), S3 (raw doc landing), a lightweight managed orchestrator (Prefect/Dagster), dbt for SQL-side transforms, hosted LLM/embedding/reranker models. **Name the heavy tools — Kafka, full self-run Airflow, a dedicated vector DB, Spark — but explicitly defer them, each with a trigger condition** (Kafka only when ingest becomes streaming; dedicated vector DB only past pgvector's limits; etc.). Naming the tool *and* its deferral is a stronger judgment signal than deploying everything. The single guiding rule: fewest datastores/services that work, split out only when a measured limit forces it — every added component is operational surface area a small team pays for continuously.
- **No over-formatting in prose deliverables.** The decision log is one paragraph, not a bulleted list.

## Normalization Module — what "elegant" means here

The module should demonstrate a layered pipeline, not a single fuzzy-match pass:

1. **Normalize surface form** — strip legal suffixes (Co., Ltd., Inc., GmbH, Limited, K.K., Co. Limited), casefold, collapse whitespace, handle known acronym↔expansion pairs.
2. **Block** — group candidate duplicates cheaply (e.g. by country + normalized-name prefix or token signature) so you don't do O(n²) fuzzy comparisons across the whole set.
3. **Match within blocks** — fuzzy similarity (token-set ratio) plus an explicit alias dictionary for cases fuzzy matching alone can't catch (`CATL` ↔ `Contemporary Amperex`). Emit pairwise matches as scored, *reasoned* edges rather than merging on the spot.
4. **Cluster (union-find)** — compute transitive closure over accepted match edges so A↔B and B↔C resolve A, B, C into one entity even without a direct A↔C match. This is the step that makes it real entity resolution instead of naive pairwise dedup, and it's the algorithmic centerpiece.
5. **Resolve conflicts field-by-field** — a per-field survivorship strategy over each cluster (revenue → highest source-trust, tie-broken by recency; legal name → longest well-formed form; certifications → union), each with a stated rule and a confidence score fed by a shared **source-trust table** (official registry > filing > company site > news > aggregator).
6. **Handle hierarchy** — detect and record parent/subsidiary rather than blindly merging them (a subsidiary is not a duplicate of its parent).

Emit a canonical record that carries field-level provenance (which source each surviving field came from), a per-field confidence value, and a merge explanation (which records fused and why) — that provenance and explanation are what make it "verified" and auditable rather than just "merged." Define the source-trust table once and have both survivorship and confidence scoring consume it. Explicitly de-scope phonetic/transliteration matching with a comment (considered, deferred) rather than silently omitting it.

## Two cross-cutting principles the whole design should reflect

**Data prioritization — spend finite budget where correctness matters most.** The system has limited crawl, verification, and compute budget. Ingest is prioritized by *value × staleness* (query demand × source trust × downstream dependencies, times how far past TTL). Fields are prioritized by *stakes*: disqualifying hard filters (country, certification) get the verification budget and demand high confidence; ranking-only soft signals accept lower confidence. Source conflicts resolve via the source-trust table. These are one principle, not three mechanisms.

**Graceful degradation — fail safe, and degrade rather than error.** Every external dependency and LLM call is a break point, and each must have a fallback that returns a usable, honestly-labeled result. Two rules: (a) *fail safe over fail merged* — for the one irreversible operation (entity merge), uncertainty defaults to not acting, because an unmerged duplicate is recoverable but a wrong merge silently corrupts; (b) *degrade, don't error* — a parser failure drops to rule-based extraction, a vector-store outage drops to structured SQL filtering, an enrichment timeout serves cached data with a stale flag. Fallbacks reuse existing primitives (confidence score, `unverified` flag, `as_of` timestamp, review queue) — resilience is not bolted on.
