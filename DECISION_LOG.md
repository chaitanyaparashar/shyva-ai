# Decision Log

**The prototype runs and verifies:** `python normalize.py` collapses **10 messy records → 8 canonical entities**, idempotently (byte-identical on re-run), and passes three load-bearing checks — the `CATL` transitive triple fuses into one entity via union-find (even though `r1↔r3` scores only 21), the `Samsung SDI`/`SDS` false-friend pair stays split, and `Panasonic Holdings` is linked `parent_of`→`Panasonic Energy` rather than merged. Given a 2-hour box I spent depth where the grade and the risk concentrate — the entity-resolution module (`normalize.py`, the one runnable artifact) — and kept the architecture broad but shallow, leaning on tables, a rendered Mermaid diagram, and a worked trace of the actual test query rather than prose. The central design stance is **LLM-proposes / rules-dispose**: models do the language-in work (query decomposition, constraint typing, unstructured→structured extraction) and language-out work (RAG synthesis, conversational refinement), but they are **deliberately excluded from the HIGH-risk core** — relationship truth ("supplies to automotive OEMs") and field survivorship (which revenue/country/cert becomes canonical, resolved via the shared **source-trust table**) stay fully deterministic, which is what makes the pipeline idempotent (re-running yields byte-identical output, verified) and every surviving value auditable back to a cited source. Model use follows a **minimal-sufficient** discipline: the smallest model that clears each stage's bar, non-LLM models where they are cheaper and better (an embedding model for blocking, a cross-encoder for re-ranking), and a **second model introduced in exactly one place** — an independent, different-family cross-checker on merge proposals — because a wrong merge is the single worst failure and merge is the only low-volume, high-stakes, irreversible decision that warrants a second opinion. Everything resilient reuses primitives already in the design (the confidence score, the `unverified` flag, the `as_of` timestamp, the human-review queue), so graceful degradation wasn't bolted on afterward — it fell out of the same parts.

### What I did NOT build, and why (deliberate cuts / startup restraint)
- **No live crawling** — supplier data is mocked in-memory in `normalize.py`; ingestion is described, not run.
- **Datastores & orchestrator described, not stood up** — Postgres/pgvector, Redis, S3, Prefect/Dagster are named with a launch-vs-scale split, not deployed.
- **Heavy infra deferred with trigger conditions** — **Kafka** (not until streaming ingest replaces batch crawls), **full self-run Airflow** (Prefect/Dagster suffices until DAG complexity demands it), a **dedicated vector DB** (pgvector holds until millions of vectors / sub-100ms p99), **Spark** (data volume doesn't warrant a distributed engine). Naming the tool *and* its trigger is the judgment signal.
- **No phonetic / transliteration matching** (Metaphone/Soundex, CJK romanization) — considered, deferred; it slots in as an extra candidate-edge source, not a rewrite.
- **No auth / multi-tenancy.**
- **Resolution runs on a mocked batch**, not a streaming ingest — the same pure functions are written to run as an offline batch job over the registry at scale.

### Severity-ranked vulnerabilities (worst first — blast radius + mitigation)
1. **Silent wrong-merge** — two real distinct suppliers collapse into one; invisible to the user, corrupts the shortlist. The single worst failure. → *Fail-safe-don't-merge*: uncertainty defaults to not merging. **The demo caught a real instance**: the stubbed LLM proposed merging `Panasonic Holdings` with `Panasonic Energy` (conf 0.66, shared brand token); the deterministic gate refused it for lack of ≥-threshold support and linked them `parent_of` instead — logged in the fail-safe log. The `Samsung SDI`/`SDS` acronym guard and the ungated-LLM-proposal path are the same principle; only ≥-threshold deterministic edges union.
2. **Stale certification shown as valid** — a decommissioned/expired cert could fail a procurement audit. → per-field TTLs + `as_of` staleness flags on the serving path.
3. **Confidently-wrong inferred/extracted signals** — even with the LLM barred from asserting relationships, extraction can err. → `unverified` labeling + cited-source gating (no span ⇒ not promoted to verified).
4. **No ground-truth dataset** — resolution accuracy is unmeasured, so threshold/blocking tuning is guesswork. → the week-one build below.
5. **Enrichment latency / rate limits** on the slow path. → *degrade-don't-error*: serve cached + backfill async, never block on a live call.
6. **Blocking recall cost** — country-blocking skips cross-block comparisons and will miss some true duplicates. → accepted recall/latency tradeoff; add a second name-signature block pass at scale.

Multi-model routing and the cross-checker are **unjustifiable without evals**, so they are architected-in but **shipped single-family at launch**. Every fallback above reuses an existing primitive (confidence, `unverified`, `as_of`, review queue) — resilience wasn't bolted on.

### First thing I'd build with a full week
A **labeled evaluation set + a golden entity registry.** This is the keystone because it converts resolution quality from vibes into a measurable number, which unlocks everything downstream: per-stage model routing and the merge cross-checker can be switched on *with evidence* of accuracy gain rather than faith, and the LLM gets a stable canonical set to classify *against* instead of re-resolving from scratch each run. It ties directly to vulnerabilities (4) and (1) — **you cannot fix a silent wrong-merge you cannot measure**, and the eval set is the instrument that makes the worst failure visible.
